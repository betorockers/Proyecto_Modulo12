Módulo 12: HERRAMIENTAS DE AUTOMATIZACIÓN DE UN FLUJO COMPLETO WEB + API + CICD 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
Capítulo 	1: 	AUTOMATIZACIÓN 	DE 	UN 	FLUJO COMPLETO WEB + API + CI/CD 
 
La automatización de un flujo completo en un entorno de desarrollo moderno comprende la validación de distintos niveles del sistema: interfaz de usuario (web), lógica de negocio (API), y procesos de integración y despliegue continuo (CI/CD). Este enfoque permite mantener la calidad del software, disminuir errores en producción y acelerar los ciclos de entrega.  
 
El presente capítulo aborda el proceso de integración de pruebas automatizadas en un flujo continuo de desarrollo y despliegue, utilizando herramientas especializadas que permiten orquestar todo el proceso desde el código fuente hasta el ambiente de producción o staging. 
 
 
Componentes del Flujo Automatizado 
 
 
Pruebas Automatizadas de la Interfaz Web 
 
Las pruebas de interfaz gráfica se enfocan en simular la interacción del usuario con el sistema. Se validan aspectos como la navegación, formularios, estados visuales y mensajes. 
 
Herramientas comunes: Cypress, Playwright, Selenium. 
 
Ejemplo de flujo de prueba: 
 
• Acceder a la página de login. 
 
• Ingresar credenciales. 
 
• Verificar redirección a panel de usuario. 
 
• Validar la visibilidad de elementos clave.

Pruebas Automatizadas de API 
 
Las pruebas de servicios API permiten validar la lógica de negocio, operaciones de base de datos, respuestas de backend y cumplimiento de especificaciones REST o GraphQL. 
 
Herramientas comunes: Postman, Newman, Rest Assured, Supertest. 
 
Pruebas típicas: 
 
• Verificación de códigos de estado HTTP. 
 
• Validación de estructuras JSON. 
 
• Pruebas de carga mínima y tiempos de respuesta.

Automatización en CI/CD 
 
La automatización continua se integra en pipelines que ejecutan todas las etapas del ciclo de vida: compilación, pruebas, revisión de calidad y despliegue. 
 
Herramientas comunes: GitHub Actions, GitLab CI, Jenkins, CircleCI. 
 
Etapas comunes de un pipeline: 
 
• Clonación del repositorio. 
 
• Instalación de dependencias. 
 
• Ejecución de pruebas unitarias. 
 
• Ejecución de pruebas UI y API. 
 
• Análisis de calidad de código. 
 
• Despliegue automático (si todas las pruebas pasan). 
 
 
 
 
 
 
 
Ejemplo de archivo de configuración para GitHub Actions: 
 
Archivo: .github/workflows/flujo-completo.yml 
 

 
 
name: Flujo completo de pruebas 
 
on: [push] 
 
jobs:   test: 
    runs-on: ubuntu-latest 
 
    steps: 
- name: Clonar código       uses: actions/checkout@v3 
 
- name: Configurar Node.js       uses: actions/setup-node@v3       with: 
        node-version: '18' 
 
- name: Instalar dependencias       run: npm install 
 
- name: Ejecutar pruebas Cypress       run: npm run cy:run 
 
- name: Ejecutar pruebas API con Newman 
      run: | 
        npm install -g newman 
        newman run ./tests/api/postman_collection.json 
 
 
 
 
 
 
 
 
 
Buenas Prácticas 
 
• Separar pruebas por tipo: Evita mezclar pruebas unitarias, de integración y de interfaz. 
 
• Usar datos de prueba aislados: Para prevenir interferencias entre ejecuciones. 
 
• Habilitar reportes automáticos: Preferir reportes en HTML o integraciones con dashboards. 
 
• Fail fast: Detener el pipeline si alguna prueba crítica falla. 
 
• Reintentar pruebas inestables: Configurar retries en pruebas UI si son frágiles. 
 
• Utilizar entornos controlados o mockeados para pruebas API. 
 
 
La automatización de un flujo completo que abarca interfaz web, servicios API y procesos CI/CD representa un componente fundamental en cualquier estrategia de aseguramiento de calidad moderna. Su implementación requiere una combinación de habilidades técnicas, uso adecuado de herramientas y una visión clara del proceso de desarrollo. Este enfoque permite no solo detectar errores de forma temprana, sino también mantener la integridad del software en ciclos de entrega cada vez más cortos y exigentes. 
 
 
 
 
 
 
 
 
 
 
 
 
 
Capítulo 2: ORGANIZACIÓN Y DOCUMENTACIÓN DEL REPOSITORIO 
 
Una correcta organización y documentación del repositorio de pruebas automatizadas es fundamental para garantizar la mantenibilidad, escalabilidad, trazabilidad y colaboración efectiva entre los miembros del equipo de desarrollo, QA y DevOps. Un repositorio bien estructurado permite a cualquier persona entender la lógica del proyecto, ejecutar las pruebas sin dificultades y aportar mejoras sin introducir errores. 
 
Este apartado entrega las recomendaciones técnicas y estructurales para organizar un repositorio orientado a pruebas automatizadas en flujos Web, API y CI/CD.

Integración CI/CD 
 
Ver configuración en .github/workflows/flujo-completo.yml 
 
El archivo .github/workflows/flujo-completo.yml forma parte de la integración continua (CI) y entrega continua (CD) del proyecto. Este archivo define el flujo automatizado que se ejecuta cada vez que ocurre un evento en el repositorio (por ejemplo, un push, pull request o release). Su objetivo es garantizar que todo el ciclo de construcción, prueba y despliegue de la aplicación esté controlado y automatizado. 
GitHub Actions permite definir estos flujos mediante archivos YAML, que especifican los pasos secuenciales a ejecutar en máquinas virtuales preconfiguradas. En este caso, el flujo cubre pruebas automatizadas para interfaz web y API, dentro de un pipeline reproducible y versionado.

Desglose y explicación de un ejemplo práctico 
 
A continuación, se muestra un archivo de configuración de ejemplo, con cada sección explicada detalladamente. 
 
name: Flujo completo de pruebas automatizadas 
 
on:   push:     branches:       - main   pull_request:     branches: 
- main 
 
jobs:   test: 
    runs-on: ubuntu-latest 
 
    steps: 
- name: Clonar repositorio         uses: actions/checkout@v3 
 
- name: Configurar Node.js         uses: actions/setup-node@v3         with: 
          node-version: '18' 
 
- name: Instalar dependencias         run: npm install 
 
- name: Ejecutar pruebas UI (Cypress)         run: npm run test:e2e 
 
- name: Ejecutar pruebas API (Newman)         run: | 
          npm install -g newman           newman run ./tests/api/postman_collection.json --reporters cli,html 
 
- name: Generar artefactos de reporte         uses: actions/upload-artifact@v3         with: 
          name: reporte-pruebas           path: reports/ 
 
Explicación de cada sección 
 
name 
Define el nombre del workflow, visible en la pestaña Actions de GitHub. 
 
name: Flujo completo de pruebas automatizadas 
 
 
on 
Define los eventos que activan el workflow. En este caso, se ejecuta: 
 
Al hacer push en la rama main. 
 
Al crear un pull_request hacia la rama main. 
 
on:   push:     branches:       - main   pull_request:     branches: 
      - main 
 
 
 
 
 
jobs 
Define los trabajos a ejecutar. En este ejemplo, hay un solo trabajo llamado test, ejecutado en una máquina virtual con Ubuntu. 
 
jobs:   test: 
    runs-on: ubuntu-latest 
 
 
steps 
Lista las acciones dentro del trabajo test: 
 
 
1.- Clonación del repositorio: 
 
- name: Clonar repositorio   uses: actions/checkout@v3 
 
 
2.- Configuración del entorno de ejecución (Node.js): 
 
- name: Configurar Node.js   uses: actions/setup-node@v3   with: 
    node-version: '18' 
 
 
3.- Instalación de dependencias: 
 
- name: Instalar dependencias   run: npm install 
 
 
4.- Ejecución de pruebas de UI (Cypress o similar): 
 
- name: Ejecutar pruebas UI (Cypress)   run: npm run test:e2e 
 
 
5.- Ejecución de pruebas de API con Postman (Newman): 
 
- name: Ejecutar pruebas API (Newman)   run: | 
    npm install -g newman     newman run ./tests/api/postman_collection.json --reporters cli,html 
6.- Carga de reportes como artefactos del workflow: 
 
- name: Generar artefactos de reporte   uses: actions/upload-artifact@v3   with: 
    name: reporte-pruebas     path: reports/ 
 
 
Buenas prácticas para workflows CI/CD 
 
• Fail fast: el workflow debe detenerse en el primer error crítico. 
 
• Paralelización: separar pruebas por tipo (UI, API, integración) en jobs diferentes si se desea mayor velocidad. 
 
• Persistencia de reportes: guardar reportes como artefactos o integrarlos con dashboards externos (Allure, ReportPortal). 
 
• Variables de entorno seguras: usar secrets de GitHub para manejar tokens o claves de entorno. 
 
• Etiquetado de versiones: automatizar releases si las pruebas son exitosas. 
 
 
 
La configuración del archivo .github/workflows/flujo-completo.yml permite establecer una automatización robusta para validar cada cambio de código mediante pruebas automatizadas. Este enfoque mejora la eficiencia del equipo, reduce el error humano, y asegura un flujo de entrega continuo alineado con los principios de DevOps y calidad de software.
